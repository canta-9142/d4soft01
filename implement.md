# 実装方針・実装上の注意点

## 1. UMLクラス図の概要

`uml.md` のクラス図は、Canvas型ToDoアプリを実装するための主要なデータ構造と責務を表している。

中心となるのは `Application` である。`Application` は現在のモード、現在選択中のキャンバス・タスク・接続線、未保存状態、コピー情報、取り消し履歴を管理し、タスク作成、編集、移動、接続、検索、深さフィルター、保存、復元などの操作を受け付ける。

永続化対象の全体状態は `AppState` にまとめる。`AppState` は保存データのバージョン、キャンバス一覧、現在のキャンバスID、表示設定を持つ。`Canvas` はタスクと接続の所有単位であり、`Task` と `Connection` は必ずいずれかのキャンバス配下に含まれる。

`Task` はタイトル、説明文、ステータス、位置、作成日時、更新日時を持つ。`Connection` は接続元と接続先を `parentTaskId`、`childTaskId` としてID参照で保持する。接続は向きを持つが、深さ計算では向きを考慮しない。

`ViewSettings` は検索文字列、ステータス絞り込み、深さフィルターの有効/無効、基準タスクID、最大深さを管理する。保存形式の `viewSettings` にも `statusFilter` を含め、ステータス絞り込みが無効な場合は `null` とする。表示対象タスクは、現在のキャンバス内のタスクに対して、この表示設定を適用して求める。

`LocalStorageService` は `AppState` の保存と復元を担当する。復元時は成功・失敗、復元後の状態、エラーメッセージを `RestoreResult` として返す。

`FilterService` はステータス、キーワード、深さによる絞り込み処理を担当する。特に深さフィルターは接続関係をグラフとして扱い、基準タスクからの最短距離を計算する。

`HistoryManager` は取り消しとやり直しの履歴を管理する。履歴は `HistoryEntry` として記録し、最大50件まで保持する。`ClipboardState` はアプリ内コピー・貼り付け用の一時データであり、localStorageへの保存対象には含めない。

`dfd.md` の Web Storage API への矢印は、保存対象データを扱う流れを示す概念図として読む。実装では、タスク登録、移動、接続作成、検索条件変更などの各操作ごとに localStorage へ直接書き込むのではなく、まずメモリ上の `AppState` を更新する。localStorage への書き込みは、保存ボタン、Ctrl+S、キャンバス切り替え前の保存処理でのみ行う。

## 2. 実装上の注意点

### 2.1. 保存データと画面状態を分ける

localStorageに保存するのは `AppState` を基本とする。`Application` が持つ `currentCanvas`、`currentTask`、`currentConnection` などは、画面操作中の派生状態であり、そのまま保存しない。

保存対象に含めるべきものは、キャンバス、タスク、接続、現在のキャンバスID、表示設定である。表示設定には検索文字列、ステータス絞り込み、深さフィルターの有効/無効、基準タスクID、最大深さを含める。コピー情報、選択中タスク、選択中接続線、接続モード中の一時的な接続元、履歴スタックは基本的に保存しない。

### 2.2. ID参照の整合性を必ず検証する

`Connection` は `Task` オブジェクトそのものではなく、`parentTaskId` と `childTaskId` を持つ。保存前と復元時には、接続が参照しているタスクIDが同じキャンバス内に存在するかを検証する。

タスク削除時は、そのタスクを参照する接続も同時に削除する。キャンバス削除時は、そのキャンバスに属するタスクと接続をまとめて削除する。

### 2.3. 深さフィルターは無向グラフとして計算する

接続線は親タスクIDと子タスクIDを持つため向きがある。ただし仕様上、深さ計算では向きを考慮しない。そのため `filterByDepth()` では、`parentTaskId -> childTaskId` だけでなく `childTaskId -> parentTaskId` も隣接関係として扱う。

循環する接続関係は許可されるため、探索済みタスクIDを記録しながら幅優先探索で最短距離を求める。複数経路で到達できる場合は、最初に到達した距離をそのタスクの深さとして扱う。

### 2.4. 検索、ステータス、深さ条件は同時に適用する

表示対象タスクは、検索条件、ステータス絞り込み、深さフィルターのすべてを満たすものにする。条件の適用順は実装上自由だが、最終結果が仕様と一致するようにする。

表示対象接続は、両端のタスクIDがどちらも表示対象タスクに含まれる接続だけにする。検索条件、ステータス絞り込み、深さフィルターによって片方でも非表示になった接続線は描画しない。

検索文字列は前後の空白を取り除き、空文字の場合は検索条件を無効として扱う。検索対象はタスクのタイトルと説明文で、大文字小文字は区別しない。

深さフィルターが無効な場合でも、`depthBaseTaskId` と `maxDepth` の入力値は保持してよい。ただし表示対象の判定には使わない。

### 2.5. 復元処理は失敗してもアプリを起動できるようにする

localStorageのデータは、手動編集や古いバージョンの保存データによって壊れている可能性がある。復元時はJSONとして読めるか、必須項目が存在するか、型が正しいか、ID参照が正しいか、ステータス値が定義済みかを検証する。

保存時と復元時の検証は `spec.md` の保存条件に合わせる。キャンバスID、タスクID、接続IDの重複、有限な座標、空でないタイトル、接続の自己参照や重複、`currentCanvasId` の妥当性、`statusFilter` の値、深さフィルター有効時の基準タスクIDと最大深さも確認する。

保存データの `version` は `1` のみを有効とする。未知のバージョン、必須項目の欠落、ISO 8601形式として扱えない日時、配列であるべき値が配列でない場合は復元失敗として扱う。未知の追加フィールドは無視してよいが、既知フィールドの型が不正な場合は初期値で補正せず復元失敗にする。

復元に失敗した場合は、保存データを上書きせず、初期状態の `AppState` を作成してアプリを起動する。その場合、`RestoreResult.success` は `false` とし、画面上では復元失敗を表示できるようにする。

復元に成功した場合は、`HistoryManager` の取り消し履歴とやり直し履歴を空にし、`Application.isDirty` を `false` に戻す。

### 2.6. Undo/Redoは操作単位で記録する

取り消し対象は、タスク登録、タスク編集、タスク移動、タスク削除、接続作成、接続削除、貼り付け、キャンバス削除である。検索条件変更、ステータス絞り込み変更、深さフィルター変更、保存、復元、キャンバス切り替えは履歴に含めない。

`HistoryEntry` には操作種別、対象ID、対象キャンバスID、変更前状態、変更後状態、同時に影響を受ける接続やキャンバスを記録する。新しい取り消し対象操作を記録したときは、redoStackを空にする。

タスク削除の履歴には、削除したタスク、そのタスクに接続していた接続一覧、削除前の選択状態、削除前の深さフィルター設定を含める。Undo時は同じキャンバスへタスクと接続を戻し、削除によって解除された深さフィルターがあれば復元する。

貼り付けの履歴には、貼り付けで作成した新規タスクの全体状態を記録する。Undo時はそのタスクと、そのタスクを参照する接続が存在する場合はそれらを削除する。Redo時は同じIDのタスクを再作成する。

キャンバス削除の履歴には、削除したキャンバス全体、削除前の `currentCanvasId`、削除前の `viewSettings`、削除前の選択中タスクID・選択中接続IDを含める。Undo時は削除前と同じ位置にキャンバスを戻し、`currentCanvasId` と表示設定も削除前に戻す。Redo時は同じキャンバスを再度削除し、仕様のキャンバス削除後ルールに従って現在のキャンバスと表示設定を更新する。

履歴はアプリ全体で最大50件まで保持する。50件を超えた場合は古い履歴から削除する。

### 2.7. 更新日時と未保存状態を忘れずに更新する

タスクのタイトル、説明文、ステータス、位置を変更した場合は、対象タスクの `updatedAt` を更新する。キャンバスのタイトルや表示位置を変更した場合は、対象キャンバスの `updatedAt` を更新する。

タスク、接続、キャンバス、表示設定に変更があった場合は、`Application.isDirty` を `true` にする。保存または復元に成功した場合は `false` に戻す。

キャンバス切り替え前に保存を実行する場合、保存に成功したときだけ `currentCanvasId` を変更する。保存条件違反またはlocalStorage書き込み失敗が起きた場合は、キャンバス切り替えを中止し、現在のキャンバスと表示設定を維持する。

### 2.8. バリデーションは保存時だけでなく操作時にも行う

タイトルが空欄または空白のみのタスクやキャンバスは作成・更新しない。同じ向き、同じ親タスクID、同じ子タスクIDの接続は重複作成しない。自己接続も作成しない。

深さフィルターでは、基準タスクが現在のキャンバス内に存在すること、最大深さが0以上の整数であることを確認する。不正な入力の場合は状態を壊さず、エラーを表示する。

ステータス絞り込みUIは、検索ウィンドウ内に検索文字列や深さフィルターと並べて配置する。選択肢は「絞り込みなし」「未着手」「進行中」「完了済み」とし、「絞り込みなし」は `statusFilter: null` として扱う。ステータス絞り込みの変更は即時に `ViewSettings` へ反映し、表示対象タスクと接続線を再計算する。

### 2.9. ユーザー入力はHTMLとして扱わない

タスクタイトル、説明文、キャンバスタイトル、検索文字列など、ユーザーが入力した文字列はHTMLとして挿入しない。DOMへ表示するときは `textContent` を使い、`innerHTML` に直接代入しない。

JSONインポートやlocalStorage復元を行う場合も、文字列を表示する前に型を確認し、想定外の値は破棄または初期値に置き換える。

### 2.10. 図にない描画層を別途整理する

UMLクラス図は主に状態とロジックを表しており、HTML要素、SVG接続線、イベントリスナー、右クリックメニュー、検索ウィンドウなどの描画実装までは詳しく表していない。

実装では、状態を変更する処理と画面を描画する処理を分ける。操作イベントは `Application` のメソッドを呼び、状態変更後に現在の `AppState` と `ViewSettings` から画面を再描画する流れにすると、保存・復元・Undo/Redoとの整合性を保ちやすい。

再描画は、現在のキャンバス取得、表示対象タスクの算出、表示対象接続の算出、タスクカードDOMの更新、SVG接続線の更新、メニューバーと検索ウィンドウの状態更新の順で行う。タスク移動中は、移動対象タスクの位置更新に合わせて接続線だけでも即時更新し、ドラッグ完了時に履歴記録と全体再描画を行う。

タスクカード、キャンバスタイトル、検索文字列、エラー表示など、ユーザー入力または保存データ由来の文字列をDOMに出す場合は `textContent` を使う。接続線はSVGで描画し、両端タスクの中心座標から線と中央矢印を計算する。非表示タスクを参照する接続線、削除済みタスクを参照する接続線、不正なID参照を持つ接続線は描画しない。
